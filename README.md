## What is Harness Engineering?

Harness Engineering is a practice framework built around AI coding agents. It solves three fundamental problems:

1. **Self-evaluation bias** вҖ” When agents evaluate their own output, they tend to confidently praise it вҖ” even when quality is clearly mediocre (Anthropic Research, 2025)
2. **Context exhaustion** вҖ” Agents try to do everything at once, often running out of context mid-task, leaving half-finished features
3. **State loss** вҖ” Across sessions, a new agent has no idea what the previous one did

## Integration with OpenSpec

### Division of Responsibilities

OpenSpec and Harness solve **different problems** and own **different stages** of the development lifecycle:

| Stage | Who Owns It | What Happens |
|-------|-------------|-------------|
| **Propose** | OpenSpec | Define what to build, scope, motivation |
| **Specs** | OpenSpec | Expand into Given/When/Then scenarios |
| **Tasks** | OpenSpec | Break specs into actionable dev tasks |
| **Apply** | **Harness** | Execute tasks with verification loop (replaces `/opsx:apply`) |
| **Verify** | OpenSpec | Final confirmation that all specs are satisfied |
| **Archive** | OpenSpec | Archive the completed change |

In short: **OpenSpec decides what to build. Harness ensures it's built correctly.**

### Where Harness Plugs In

Harness inserts itself at exactly one point вҖ” between **Tasks** and **Verify**:

```
/opsx:propose в”ҖвҶ’ /opsx:specs в”ҖвҶ’ /opsx:tasks в”ҖвҶ’ /project:harness-apply в”ҖвҶ’ /opsx:verify в”ҖвҶ’ /opsx:archive
       в”Ӯ               в”Ӯ              в”Ӯ                   в”Ӯ                     в”Ӯ               в”Ӯ
   OpenSpec         OpenSpec       OpenSpec            Harness              OpenSpec         OpenSpec
                                      в”Ӯ                   в”Ӯ
                                      в–ј                   в–ј
                                  tasks.md в”Җв”Җв”Җв”Җв”Җв”ҖвҶ’ feature_tests.json
                                  specs.md в”Җв”Җв”Җв”Җв”Җв”ҖвҶ’ (extracts scenarios)
```

### What Harness Reads from OpenSpec

Harness reads **two files** generated by OpenSpec:

| File | What Harness Extracts |
|------|----------------------|
| `changes/<id>/tasks.md` | Task list вҖ” each `- [ ]` item becomes a task in `feature_tests.json` |
| `changes/<id>/specs.md` | Spec scenarios вҖ” each Given/When/Then becomes a `spec_scenario` to verify against |

Example вҖ” OpenSpec generates:

```markdown
<!-- specs.md -->
## User Registration
- Given valid email and password, When POST /auth/register, Then return 201

<!-- tasks.md -->
- [ ] 1.1 Create User model with password hashing
- [ ] 1.2 Add /auth/register endpoint
```

Harness transforms this into:

```json
{
  "tasks": [
    {
      "id": "1.2",
      "description": "Add /auth/register endpoint",
      "spec_scenarios": ["Given valid email and password, When POST /auth/register, Then return 201"],
      "verification_commands": ["pytest tests/api/test_auth.py::test_register -v"],
      "passes": false
    }
  ]
}
```

### What Harness Produces Alongside OpenSpec

After harness-apply completes, the change directory contains **both** OpenSpec and Harness artifacts:

```
changes/add-user-auth/
в”ңв”Җв”Җ proposal.md              вҶҗ OpenSpec (unchanged)
в”ңв”Җв”Җ specs.md                 вҶҗ OpenSpec (unchanged)
в”ңв”Җв”Җ tasks.md                 вҶҗ OpenSpec (unchanged)
в”ңв”Җв”Җ feature_tests.json       вҶҗ Harness (generated, tracks pass/fail per task)
в”ңв”Җв”Җ claude-progress.txt      вҶҗ Harness (progress state for session recovery)
в””в”Җв”Җ evaluations/             вҶҗ Harness (evaluation reports per task)
    в”ңв”Җв”Җ task-1.1-attempt-1.md
    в”ңв”Җв”Җ task-1.2-attempt-1.md    (failed)
    в””в”Җв”Җ task-1.2-attempt-2.md    (passed after fix)
```

When you run `/opsx:archive`, all of these are archived together вҖ” the evaluation reports serve as **quality evidence** for the implementation.

### Why Not Modify OpenSpec Directly?

1. **Separation of concerns** вҖ” "what to build" and "how to build reliably" are orthogonal
2. **Maintenance cost** вҖ” OpenSpec has 23 tool adapters and is actively evolving; a fork means constant rebasing
3. **Minimal footprint** вҖ” Harness is 4 markdown files vs. maintaining a full TypeScript project
4. **Composability** вҖ” Harness works with or without OpenSpec (you can write `feature_tests.json` by hand)

## Hooks: Automated Triggers (Inspired by ralph-wiggum)

Pure markdown commands rely on Claude following instructions in-context. Hooks add **deterministic enforcement** вҖ” they run as shell scripts at specific lifecycle events, regardless of what Claude "decides" to do.

This approach is inspired by [ralph-wiggum](https://github.com/anthropics/claude-code/tree/main/plugins/ralph-wiggum), Anthropic's official plugin that uses a **Stop hook** to prevent Claude from exiting until a task is truly complete.

### Three Hooks

| Hook | Event | What It Does |
|------|-------|-------------|
| `stop-check.sh` | **Stop** | Checks `feature_tests.json` вҖ” if any task has `passes: false`, **blocks Claude from stopping** and tells it which tasks remain |
| `session-init.sh` | **SessionStart** | Reads `feature_tests.json` + `claude-progress.txt` and prints a status summary so Claude immediately knows where to resume |
| `post-tool-notify.sh` | **PostToolUse** (Bash) | After a `git commit`, reminds Claude to launch the evaluator subagent |

### How the Stop Hook Works

This is the key mechanism. When Claude thinks it's "done" and tries to exit:

```
Claude: "All done! I've implemented the feature."
         вҶ“ (tries to stop)
Stop hook runs вҶ’ reads feature_tests.json
         вҶ“
         в”ңв”Җв”Җ All tasks passes: true  вҶ’ {"decision": "allow"}  вҶ’ Claude exits вң…
         в”Ӯ
         в””в”Җв”Җ Some tasks passes: false вҶ’ {"decision": "block",  вҶ’ Claude CANNOT exit вқҢ
                                          "reason": "3/7 tasks passed. Remaining:
                                            - Task 2.2: Add JWT middleware
                                            - Task 3.1: Add token refresh
                                          Continue working."}
                                                вҶ“
                                         Claude sees the reason
                                         and continues working
                                                вҶ“
                                         (cycle repeats)
```

This solves the **self-evaluation bias** problem mechanically: Claude can't just *say* it's done вҖ” the hook independently verifies by checking `feature_tests.json`.

### Hooks vs Pure MD

| Aspect | Pure MD (commands only) | MD + Hooks |
|--------|------------------------|------------|
| Evaluation trigger | Claude follows prompt instructions | `post-tool-notify.sh` reminds after every commit |
| Exit control | Claude decides when to stop | `stop-check.sh` blocks exit until verified |
| Session recovery | Claude reads files if prompted | `session-init.sh` auto-prints status |
| Reliability | Depends on context window & attention | Deterministic shell scripts, always run |

**Recommendation**: Use hooks for production workflows. Use pure MD for simple or experimental tasks.

## Dependencies

**Plugin install** (Method 1): No dependencies. Pure markdown files using Claude Code's built-in agents/commands/hooks.

**CLI install** (Method 2 / npm): Requires Node.js >= 20. The CLI has runtime dependencies (commander, zod, yaml, chalk) installed automatically via npm. Playwright MCP is optional (only for L4/L5 browser testing).

## Usage

```bash
# 1. Create proposal with OpenSpec (unchanged)
/opsx:propose "add-user-auth"
/opsx:specs
/opsx:tasks

# 2. Use harness-apply instead of /opsx:apply  вҶҗ the only change
/project:harness-apply add-user-auth

# 3. Verify and archive (unchanged)
/opsx:verify
/opsx:archive
```

What `/project:harness-apply` does automatically:

- Converts tasks.md into verifiable `feature_tests.json`
- Executes tasks **one at a time** (not all at once)
- Launches an **independent evaluator subagent** after each task (not self-judging)
- Auto-launches fixer subagent on failure (max 3 retries)
- Maintains progress files for **cross-session recovery**

Resume after session interruption:

```bash
/project:harness-apply add-user-auth
# Detects existing feature_tests.json вҶ’ resumes from last incomplete task
```

## Use Without OpenSpec

You can skip OpenSpec entirely. Just create `feature_tests.json` manually:

```json
{
  "change_id": "my-feature",
  "tasks": [
    {
      "id": "1",
      "description": "Add health check endpoint",
      "spec_scenarios": ["GET /health returns 200 with status ok"],
      "verification_commands": ["pytest tests/test_health.py -v"],
      "passes": false,
      "evaluation_attempts": 0
    }
  ]
}
```

Then run:

```bash
/project:harness-apply my-feature
```

## Core Components

| Component | File | Role |
|-----------|------|------|
| **Spec Reviewer** | `.claude/agents/spec-reviewer.md` | **Reviews and strengthens specs BEFORE test generation** |
| **Initializer Agent** | `.claude/agents/initializer.md` | **Generates verification material from reviewed specs** |
| Evaluator Agent | `.claude/agents/evaluator.md` | Verifies implementation in independent context |
| Fixer Agent | `.claude/agents/fixer.md` | Minimal code fixes based on evaluation reports |
| Harness Apply | `.claude/commands/harness-apply.md` | Orchestrates the full init вҶ’ code вҶ’ evaluate вҶ’ fix loop |
| Progress Tracker | `.claude/skills/progress-tracker/SKILL.md` | Manages feature_tests.json and progress state |

### Verification Levels

Different tasks need different verification methods. See [docs/verification-strategies.md](docs/verification-strategies.md) for full details.

| Level | Tools | Best For |
|-------|-------|----------|
| **L1: Static** | ruff, mypy, tsc | All code (lint + type check) |
| **L2: Unit** | pytest, jest | Business logic, models, utilities |
| **L3: Integration** | pytest + curl/httpx | API endpoints, database operations |
| **L4: E2E/Browser** | **Playwright MCP** | UI interactions, frontend features |
| **L5: Visual** | Playwright screenshots | Styling, layout, design verification |

**Key insight**: In TDD mode (`tdd_mode: true`), the initializer generates test skeletons *before* coding begins. The Coding Agent only writes implementation вҖ” tests come from specs, not from the agent. This prevents the "student writes their own exam" problem.

**For L4 (Browser)**, the evaluator uses [Playwright MCP](https://github.com/anthropics/playwright-mcp) in black-box mode вҖ” it only has browser tools (navigate, click, type, screenshot), no code access. It tests like a real user.

## Relationship with OpenSpec

This project incorporates core modules from [OpenSpec](https://github.com/Fission-AI/OpenSpec) (v1.2.0, MIT License) by [Fission AI](https://github.com/Fission-AI):

| From OpenSpec (kept) | From Harness (added) |
|---------------------|---------------------|
| Artifact Graph (DAG dependencies) | Spec Reviewer (Phase 0) |
| Delta Spec merge system | Verification Initializer (Phase 1) |
| Schema resolution (3-tier) | Harness Apply loop (Phase 2) |
| Archive & validation | Evaluator + Fixer agents |
| Instruction loader | Hooks (stop, session, post-commit) |
| Claude Code adapter | Multi-level verification (L1-L5) |

We extracted ~3,700 lines of OpenSpec's core infrastructure (~17% of the codebase) and built the harness engineering layer on top. See [UPSTREAM.md](UPSTREAM.md) for version tracking and sync instructions.

**OpenSpec focuses on "what to build" (spec-driven workflow). Harness Spec adds "how to build reliably" (verified, recoverable, self-healing execution).**
